# 低照度增强数据微调与模型交付设计

日期：2026-07-16  
状态：设计已确认，待书面规格审核

## 1. 目标

以现有最佳模型 `logs/0209_1509_lr_1e-04_b_64/best_model.pth` 为权重初始化，在已合并低照度增强样本的正式训练集 `datasets/newdata/train` 上进行 50 Epoch 微调。训练完成后保存最佳/最新 PyTorch 权重、TensorBoard 日志、逐 Epoch 摘要、ONNX 模型、评估结果和新旧模型对比报告。

训练集固定为 4918 组，验证/测试集固定为 867 组。本轮不得覆盖旧模型、旧 ONNX 或旧训练日志。

## 2. 已确认的基线

- 旧训练参数：200 Epoch、Batch Size 64、Adam、学习率 `1e-4`、权重衰减 `1e-5`；
- 学习率调度：Cosine Annealing，线性 Warmup 100 iterations；
- 网络：ResNet18、输入 `1×3×288×384`、辅助分割开启；
- 输出：50 个网格加背景类、18 个行锚点、1 条水带；
- 原有数据增强：随机旋转、上下偏移、左右偏移和水平翻转；
- 旧最佳模型：Epoch 196，验证损失 `0.19080511996357277`；
- 当前设备：NVIDIA GeForce GTX 1050，显存 2048 MiB；
- 新增低照度数据：60 组增强 JPG 与中心线 PNG，其中 59 个正样本、1 个审核负样本。

## 3. 训练方式

### 3.1 权重初始化

仅从旧 `best_model.pth` 加载形状完全匹配的模型权重，不恢复旧 Adam 状态、旧学习率或旧 Epoch。原因是旧检查点处于 200 Epoch 余弦调度末端，恢复其优化器会带入接近零的学习率，不适合作为新微调任务的起点。

加载过程必须检查缺失键、额外键和形状不匹配；除辅助头导出差异外，训练模型要求所有参数严格匹配。训练前记录旧权重 SHA-256。

### 3.2 保持不变的核心策略

- Adam 优化器；
- 权重衰减 `1e-5`；
- Cosine Annealing；
- 线性 Warmup 100 次优化器更新；
- ResNet18、辅助分割、损失权重、输入尺寸、灰度三通道预处理；
- `griding_num=50`、`row_anchor=18`、`num_lanes=1`；
- 原有几何数据增强和 867 组验证集。

### 3.3 明确修改的参数

1. 初始化由 ImageNet 预训练改为旧 `best_model.pth` 的模型权重；
2. Epoch 从 200 改为 50；
3. 初始学习率从 `1e-4` 改为 `1e-5`；
4. 物理 Batch Size 64 改为显存探测得到的最大安全 micro-batch，按 `8、4、2、1` 依次测试，预计为 2 或 4；
5. 使用梯度累积，使 `micro_batch × accumulation_steps = 64`；若 micro-batch 不能整除 64，则拒绝启动训练；
6. 开启 CUDA Automatic Mixed Precision，降低显存占用；模型结构、输入输出和损失定义不变；
7. DataLoader workers 从 8 调整为 2，降低当前设备的内存和 Windows 多进程压力；
8. 每个 Epoch 完整遍历 4918 个正式训练样本，并额外加入新增 60 个低照度索引两次，使低照度样本每轮共出现 180 次，总采样数为 5038；
9. 每次优化器更新前对累计损失除以 `accumulation_steps`，Epoch 尾部不足一个累积窗口时也必须执行一次更新；
10. 学习率调度器只在真实优化器更新时推进，调度总步数按 50 Epoch 的优化器更新数计算。

## 4. 低照度数据使用方式

训练器直接读取已复原的 60 张增强图，不在训练期间重复执行 Retinex-LAB，以避免每个 Epoch 得到不一致的输入并增加 CPU 开销。亮度提升、细节恢复、高光保护和局部对比度结果已经固化在训练 JPG 中。

低照度样本使用确定性三倍曝光：所有普通样本每轮至少出现一次，60 个低照度样本额外出现两次，然后对完整索引序列进行随机打乱。这不会删除或替换原始样本。随机种子、采样数量和低照度主干列表写入本次训练配置。

## 5. 显存预检与失败处理

正式训练前使用真实模型、辅助损失和一批真实数据执行 forward、loss、backward 的显存探测。候选 micro-batch 顺序为 8、4、2、1；每次 OOM 后清空梯度与 CUDA 缓存，再测试更小值。选择第一个连续两次通过且峰值显存留有安全余量的值。

如果 batch 1 仍失败，训练停止并报告，不关闭辅助分割、不缩小输入尺寸，也不静默改变模型。训练中若发生 OOM、NaN/Inf、数据加载错误或磁盘写入失败，保存故障信息并停止；不得自动跳过错误样本。

## 6. 日志、命名和产物

沿用原命名格式创建：

```text
logs/MMDD_HHMM_lr_1e-05_b_<microbatch>/
```

目录至少包含：

```text
best_model.pth
latest_model.pth
cfg.txt
events.out.tfevents.*
train_epoch_summary.txt
training_progress.jsonl
training_result.json
comparison_report.md
resnet_288_384.onnx
resnet_288_384_best.onnx
```

`cfg.txt` 必须同时记录 micro-batch、梯度累积步数、有效 Batch 64、AMP、低照度曝光倍数、旧权重路径与 SHA-256。根目录现有 `resnet_288_384.onnx` 和旧日志目录保持不变。

最佳模型按最低验证损失保存；最新模型每个 Epoch 覆盖本次目录中的 `latest_model.pth`。检查点保存 Epoch、模型、优化器、GradScaler、最佳验证损失和完整训练参数，以便中断后准确恢复。

## 7. 进度与时间预估

完成首个训练 Epoch 和验证 Epoch 后，以实际耗时建立初始 ETA。后续使用最近 3 个完整 Epoch 的移动平均更新时间预估，避免单轮 I/O 抖动造成大幅跳变。

训练期间在以下时点向用户报告：

- 显存预检完成和训练正式启动；
- 第 1 个 Epoch 完成；
- 此后每 5 个 Epoch，或至少每 30 分钟一次；
- 最佳验证损失更新时；
- 异常、中断或训练完成时。

每次报告包含当前 Epoch、训练/验证损失、Top-1/Top-2/Top-3、已运行时间、单轮平均耗时和预计剩余时间。

## 8. 新旧模型比较

训练前后使用完全相同的预处理和评估代码执行：

1. 原 867 组测试集：比较验证损失、Top-1、Top-2、Top-3 和辅助分割 IoU；
2. 新增 60 组低照度数据：比较分类指标和中心线位置误差，作为针对性诊断；
3. ONNX 与 PyTorch：使用固定样本检查输出形状和最大绝对误差；
4. 记录模型文件大小、最佳 Epoch 和推理耗时。

60 组低照度图已参与训练，其结果不得描述为独立泛化性能。正式总体对比以未参与训练的 867 组测试集为主，低照度对比只用于确认新增场景的拟合变化。

## 9. ONNX 导出

从本次 `best_model.pth` 导出无辅助头推理模型，参数保持：输入 `1×3×288×384`、输出 `(51,18,1)`、opset 13、动态 Batch 维。ONNX 文件先保存到本次日志目录，再用 `onnx.checker` 校验，并使用 ONNX Runtime 与 PyTorch 对同一输入做数值一致性检查。

同时保留标准名 `resnet_288_384.onnx` 和明确来源的 `resnet_288_384_best.onnx`；两者在本次目录内内容一致，不覆盖根目录旧文件。

- 显存预检通过，实际参数与 `cfg.txt` 一致；
- 50 个 Epoch 全部完成，或明确记录不可恢复的失败原因；
- `best_model.pth`、`latest_model.pth` 可重新加载；
- 训练和验证指标均有 50 条完整 Epoch 记录；
- 正式测试集保持 867 组且未被修改；
- ONNX 结构与数值校验通过；
- 新旧模型对比报告明确列出改善、退化和低照度结果的局限性；
- 所有产物仅写入新日志目录；
- 训练结束后向用户汇报参数修改、总耗时、最佳 Epoch、指标差异和全部文件位置。
